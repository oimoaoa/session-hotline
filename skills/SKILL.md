# 핫라인 — 다른 세션에 직통 질문

이 세션(너)이 오케스트레이터다. 전송·안전 가드는 검증된 스크립트가 담당한다:
`HOTLINE="$HOME/.session-hotline/hotline.sh"`

**북극성**: 사용자가 지정한 *바로 그* 세션에서 믿을 수 있는 답을 받아온다. "명령이 돌아감"은 완료가 아니다 — 엉뚱한 세션에 안 가고, 남의 세션 기록 안 망치고, 실패는 정직히 보고해야 완주다.

## 0. 자기 정체 + 첫 사용 점검

- 내가 누구인가: env `CLAUDECODE`가 있으면 **나=Claude Code**, 없으면(Codex 등) **나=Codex**로 판단.
- 이 스킬을 이 세션에서 처음 쓰거나 뭔가 이상하면: `bash "$HOTLINE" doctor` — ❌가 나오면 **멈추고 그대로 보고**(Claude Code/Codex 업데이트로 내부 포맷이 깨졌을 수 있음 — 사용자 잘못 아님).
- **나=Codex에서 CLI 호출 시**: `ask codex`·`ask claude` 둘 다 **sandbox 안에서는 실패한다**(실측 4회 — `~/.codex/state_*.sqlite` readonly, `~/.claude` 접근 차단). sandbox 안에서 먼저 시도해볼 필요 없이, **처음부터 사용자에게 "sandbox 밖 실행 승인이 필요해(정상 절차)"라고 알리고 승인받아 밖에서 실행**한다. 우회 금지, 막히면 정직 보고.

## 1. 경로 결정 (발신=나 / 수신=대상)

**용어**: 사용자에게 말할 땐 "**대화방**"으로 통일한다(클로드=세션, 코덱스=스레드 — 기술 용어는 괄호로만).

| 대상 \ 발신 | 나=Claude Code (데스크탑 앱) | 나=Claude Code (터미널 — ccd MCP 없음) | 나=Codex (앱 — codex_app 도구 있음) | 나=Codex (CLI) |
|---|---|---|---|---|
| **Codex 대화방** | `ask codex` (CLI — 원본 기록에 남음) | `ask codex` (동일) | 🌟 **native**: `codex_app.send_message_to_thread` (**원본 전달** — UI에 바로 보임) | `ask codex` |
| **Claude 대화방** | 🌟 **native**: `send_message` (MCP — **원본 전달**, §4B) | `ask claude` (CLI — **fork 자문**, 전달 불가 정직 안내) | `ask claude` (fork 자문) | `ask claude` (fork 자문) |

- 발신 환경 판별: `CLAUDECODE` env 있으면 Claude Code — 그중 **ccd_session_mgmt MCP가 있으면 데스크탑 앱, 없으면 터미널**(실측: 터미널 claude엔 이 MCP가 아예 없음). Codex는 **codex_app 도구(`list_threads`·`send_message_to_thread`)가 보이면 앱, 없으면 CLI**. ⚠️ env는 부모 프로세스에서 상속될 수 있다(실측: 클로드가 띄운 codex에 `CLAUDECODE=1`이 새어 들어감) — env와 도구 유무가 **모순되면 도구 유무를 우선**.
- **⛔ 같은 앱이면 `ask`/fork를 *시도조차 하지 마라*** (2026-07-07 오라우팅 사건 교훈): 발신이 **데스크탑 앱**(Claude Code 데스크탑 / Codex 앱)이고 대상이 **같은 앱의 세션**이면 무조건 native(`send_message`/`send_message_to_thread`). **`send_message`는 dormant(과거·종료된) 세션도 깨워 응답시킨다**(실증) — "CLI가 동기라 낫다"는 fork 정당화는 **환상**이고, 같은 앱에서 fork로 얻는 실익 0 + 원본 화면에 안 보임 + 불필요한 스크래치 세션만 남긴다. `ask`/fork·CLI 브릿지는 오직 ⓐ **앱 경계 넘기**(클↔코) ⓑ **native 도구 없는 터미널 발신**, 이 둘만.
- **같은 앱 안에서는 native 도구(🌟)가 항상 우선** — 원본 대화방 화면에 바로 보이고 왕복도 된다. `hotline.sh`는 **앱 경계를 넘거나 native 도구가 없는 환경**을 위한 다리(bridge)다.
- **코덱스→코덱스 native 사용법**(실측 2026-07-06): `list_threads`로 대화방 검색 → echo-back 확인 → `send_message_to_thread`. 수신 쪽 UI에 "다른 스레드의 Codex가 보냈습니다" 라벨로 표시됨. **왕복 답장**을 원하면 메시지에 발신 threadId(`source_thread_id`)와 "이 thread로 send_message_to_thread로 답장해줘"를 명시(그래야 수신 대화방이 발신 대화방으로 직접 회신 — 단순 확인만이면 `read_thread`로 읽어와도 됨).

- **두 가지 의미를 절대 섞지 마라** (실사용 혼동 사례 2026-07-05):
    - **fork 자문** (`ask claude`) = 대상 세션의 맥락을 복사한 read-only **분신(fork)이 답한다**. 원본 세션 화면엔 질문도 답도 **안 보이고**, 원본(을 보는 사용자)은 이 대화의 존재를 모른다. "지식 꺼내오기" 용도.
    - **원본 전달** (`send_message`) = 원본 세션 화면에 메시지가 실제로 도착한다. "전달·협업" 용도.
- **사용자 요청 해석 규칙**: "물어봐 / 답 받아와" → fork 자문 OK. "**보내줘 / 전달해줘 / 그 세션이 보게 해줘**" → 원본 전달이 필요 — 나=Claude면 send_message로, **나=Codex는 Claude 대화방으로의 원본 전달만 현재 불가**(ccd MCP 없음 — Codex→Codex는 native `send_message_to_thread`로 가능)를 정직히 말하고 대안 제시(파일로 남겨 사용자가 전달, 또는 원본 세션 쪽에서 역방향 핫라인). fork 자문으로 하고 "전송 완료"라 보고하는 것 금지.
- **`ask claude`(fork) 부적합 의도 — 실행 *전에* 멈추고 설명(공개 핵심, 2026-07-06 연속성 테스트 교훈)**: `ask claude`는 원본에 안 남는 **1회성 read-only fork**라 "대상 세션의 *기존* 지식 꺼내오기"에만 맞다. 사용자 요청에 아래 의도가 보이면 **자문을 실행하기 전에** 부적합성을 설명하고 확인받는다 — 실행부터 하면 fork가 *현재 입력*이나 원본의 *기존* 맥락으로 그럴듯하게 답해 "가짜 성공"처럼 보인다(실제 사고):
    - **연속·기억**("방금·아까·이전·기억나·이어·계속·후속·핑퐁") → fork는 이전 핫라인 호출을 기억 못 하고, 원본의 *기존* 맥락을 "방금"으로 잘못 답할 수 있다. → "이 경로론 이전 문답을 못 이어가 — 원본 세션에서 직접 잇거나 native 경로(코→코)를 써야 해."
    - **원본 전달·화면 표시**("전달·남겨·그 세션이 보게·화면에 보이면·도착하면·메시지 보이면") → fork는 *현재 입력*만 보고 "보인다"고 답할 뿐, **원본 화면 도착 증거가 아니다**. → "fork는 원본 화면에 안 남겨 — 원본 전달은 클→클 send_message에서만 가능."
    - **답장·왕복**("답장 오면·왕복") → fork는 왕복 불가 → native 경로(코→코 `send_message_to_thread`) 안내.
- **완료 보고 용어 강제**: fork로 답 받았으면 "**사본(fork) 자문 완료** — 원본 세션엔 안 보임" / send_message면 "원본 전달됨". "전송 완료" 같은 모호한 표현 금지.
- 클로드→클로드가 send_message로 **고정**인 이유: 사용자 요구로 확정된 사항 — 매 전송이 승인 게이트를 거치고 양쪽 세션 화면에 왕복이 남는다. "CLI가 동기라 낫다"고 임의로 `ask claude`로 대체하지 말 것(요구 변경은 사용자 승인부터).

## 2. 대상 세션 해석 + echo-back 확인 (필수 — 생략 금지)

- **id 직접지정이 primary** — 사용자가 UUID/id를 주면 이름 검색을 건너뛰고 그걸 쓴다. 이름은 convenience.
- **세션 ID로 지정하기 (id만 줘도 됨)** — 인계·핸드오프 문서에 적어둔 세션 주소(id)를 그대로 줘도 작동한다. **ID를 받으면 → resolve로 세션명을 역조회 → echo-back "이 대화방 맞아?" 확인 → 전송**(ID만으로 조용히 쏘지 말 것 — 붙여넣은 주소가 낡거나 오타일 수 있다). 3종 ID 전부 resolve가 세션명을 역조회한다(실측):
    - **클로드 `local_…`** (데스크탑 send_message 주소): `resolve claude "local_…"` → 세션명·uuid·cwd. **데스크탑 클로드 대상 발송의 주 경로**(사용자는 보통 이 주소를 준다).
    - **클로드 UUID** (`xxxxxxxx-…`, jsonl 파일명): `resolve claude "<UUID>"` → 세션명·**local_id**·cwd. resolve가 local_id로 변환해주니 UUID만 알아도 send_message 발송 가능.
    - **코덱스 `019f…`** (threadId): `resolve codex "019f…"` → 세션명·rollout.
- **verbatim 규칙(중요 — 실사고 2건)**: resolve에 넘기는 대화방 이름은 사용자가 준 문자열 **그대로**. 단어를 떼거나(앞의 주제어처럼 보여도!) 바꾸거나 줄이지 않는다. 사용자가 따옴표(`'…'`·`"…"`·`「…」`)로 감쌌으면 그 안쪽이 이름(도구가 따옴표는 알아서 벗김).
- CLI 경로: `bash "$HOTLINE" resolve claude "<제목|UUID>"` 또는 `resolve codex "<이름|ID>"` — 출력은 `{match_type, candidates, suggestions}`:
  - `match_type`: exact(완전일치) / substring(부분일치) / token(단어 전부 포함) / uuid / none. **부분·token 매칭이어도 echo-back 확인은 동일하게 필수.**
  - ⚠️ **rc(0/1/2)는 실패 신호가 아니라 후보 *개수* 신호다** — resolve를 `set -e`나 `&&` 체인 안에서 부르면 rc=2(복수 후보)를 에러로 오인해 멈춘다(2026-07-12 리허설: 여러 세션이 실제로 오해). rc를 변수로 받아 분기하라.
  - rc=0(1건): echo-back 확인으로. `match_type`이 exact가 아니면 그 사실도 언급("부분일치로 찾음").
  - rc=1(0건): `suggestions`(비슷한 후보)가 있으면 "혹시 이거?"로 제시, 없으면 "못 찾음" 정직 보고. **임의로 단어를 바꿔 재검색하지 않는다** — 순서는 ①사용자 원문 재확인 ②suggestions 제시뿐.
  - rc=2(복수): 후보 전부 보여주고 고르게 한다. **완전히 같은 제목이 여러 개**면 보통 이어하기·요약으로 기록 파일이 갈라진 것 — 최신 수정본이 현재 대화방일 가능성이 높다고 안내하고 사용자 확인.
  - `suggestions`에 유사 제목(예: "X"와 "X 2")이 있으면 echo-back에 함께 표시(오지목 방지).
  - `cwd_folder_match: false`면 진행하지 말고 보고(내부 포맷 변화 가능성).
- **다중 발송**(여러 방에 동시): 기본은 **한 방 택1**(echo-back 확인). 사용자가 **명시적으로** "여러 방에/다 물어봐"라고 할 때만 대상 전부를 echo-back으로 나열해 확인받고 **각각 전송**한다(send_message·ask는 방마다 한 번씩, 각 전송은 개별 승인 게이트). 사용자가 다중을 요청하지 않았는데 이름이 복수로 매치되면 임의로 다 보내지 말고 택1 확인.
- **grep/rg로 대화방을 찾지 않는다** — 본문에 문자열이 나온 것과 제목인 것은 다르다(실사고: 엉뚱한 세션 fork까지 감). echo-back 제목이 사용자가 말한 이름과 다르면 **중단**.
- **자동 생성 제목도 이제 resolve로 찾힌다**(2026-07-06 해결): `resolve claude`가 데스크탑 앱 저장소를 병합 검색해 자동 제목·`local_id`(send_message 주소)·보관 여부까지 제공한다. Codex 자동 이름도 원래 찾힘(session_index). 출력의 `resumable: false`면 CLI resume 불가(기록 파일 미확인) — echo-back에 표시. 데스크탑 저장소가 없는 환경에선 자동 제목만 못 찾는다(doctor가 알려줌) — 그땐 "제목 한 번 수정해주면 찾을 수 있어요" 안내.
- **이름 없는 세션(터미널에서 제목 안 지은 세션) — 폴더명+미리보기로 확인**: 저장된 제목이 없어 이름 검색이 0건인 세션이다(터미널 유저에게 흔함 — 데스크탑 앱은 자동 제목이 붙어 덜함). 이때 `resolve claude`가 자동으로 **cwd 폴더명 매칭**으로 폴백한다(`match_type: "cwd"`) — 후보에 `display_name`(폴더명)·`preview`(첫 메시지 60자)가 실린다. **echo-back에 폴더명+preview를 보여주고 확인받는다**(제목이 없으니 이 둘이 유일한 단서 — CLI `/resume` 피커가 폴더명으로 보여주는 것과 같은 방식). `cwd_truncated`가 0보다 크면(=그 값이 **생략된 세션 수**, "총 개수"가 아님) "그 폴더에 세션이 더 있어 — 최근 8개만 보여줬고 N개는 생략됐어, 더 좁혀줄래?"라고 안내(한 폴더에 세션이 몰릴 수 있음). ⚠️ preview는 **표시·확인용일 뿐 매칭 키가 아니다**(본문 grep 금지 원칙 유지 — 쿼리는 폴더명에만 매칭). 보관(archived)·삭제 세션은 폴백에서 제외된다. (Codex는 스레드 이름이 항상 있어 이 문제가 없다 — Claude 전용 처리.)
- **내 제목 알아내기**(템플릿 서명용): `bash "$HOTLINE" resolve claude "$CLAUDE_CODE_SESSION_ID"` — 내 세션의 현재 제목이 나온다(실측). Codex 앱이면 자기 threadId를 안다.
- **나=데스크탑 클로드 & 대상=Claude, 이름으로 검색할 때는 `list_sessions`(MCP)가 1차 검색원** — resolve(jsonl 파서)로 먼저 찾지 마라. 이유 둘: ①send_message의 `local_…` id는 파일명 UUID와 완전 별개 체계 ②**자동 생성 제목은 list_sessions엔 보이지만 jsonl엔 없다**(실사고: resolve로 먼저 찾다 0건 — list_sessions엔 있었음). **단 사용자가 ID(`local_…`·UUID)를 준 경우는 예외** — `resolve claude "<id>"`가 세션명·local_id·cwd를 정확히 역조회하므로(실측) resolve를 쓴다(이름의 모호성이 없어 역조회가 더 정확·빠름). resolve는 그 외 CLI 경로(코덱스 대상·fork 자문)에서.
- **echo-back 확인(찾은 세션 정보를 사용자에게 되읽어 확인받기 — 1건이어도 필수)**: `→ 이 세션 맞아? 제목:"…" / id:앞8자 / cwd:… / 마지막 수정:…` — 사용자 확인 후에만 전송. **대상이 Codex면 한 줄 추가**: "질문·답변이 그 세션의 내부 기록에 append돼(fork 불가). 단 Codex 앱 화면엔 바로 안 보일 수 있고, 앱을 껐다 켜야 보일 수 있어(실측)." **대상이 Claude이고 CLI 경로면 한 줄 추가**: "원본 세션에 메시지를 남기지 않고 read-only 분신(fork)이 답하는 방식이야 — 원본 화면엔 안 보여. 이대로 진행할까?" **유사 제목 세션**(예: "X"와 "X 2")이 목록에 보이면 echo-back 확인에 그 존재도 함께 표시해 오지목을 차단(실사용에서 실제로 느껴진 리스크).
- 해석 결과의 mtime이 최근(~10분)이면: "그 세션이 지금 열려있을 수 있어 — Codex 대상은 원본 기록에 append돼서 충돌 위험 있음(Claude 대상은 fork라 안전). 진행할까?" 확인.

## 3. 질문 준비 (정돈 → 감싸기 → 저장 → 스캔 — 이 순서 고정)

1. **먼저 사용자 질문을 정돈한다** (핵심 — 초보가 대충 말해도 되게 하는 지점. 사용자는 대충 던지고, 대상 대화방이 알아듣게 다듬는 건 오케스트레이터인 나의 일):
   - 사용자가 짧게·대충·구어로 물어도, 대상 대화방이 **맥락 없이 알아듣게** 다듬는다: ⓐ 지금 이 대화방에서 오간 관련 맥락이 있으면 한두 줄 앞에 얹고("결제를 A사로 정한 상황인데…" 식 — 이 예시 이름을 실제 질문에 쓰지 말 것) ⓑ 무엇을 묻는지 분명한 한 문장으로. **단, 사용자가 준 에러 로그·코드·구체 데이터는 축약·유실 없이 그대로 포함한다**(이미 명확한 질문은 억지로 다시 쓰지 않는다).
   - **왜곡 금지**: 사용자 의도를 바꾸거나 없는 정보를 지어내지 않는다 — 대충 말한 걸 *구체화*만 한다.
   - **불명확할 때 — 되묻기 vs 넘기기 구분**(핵심 — 사용자를 막다른 길로 몰지 않기): ⓐ **발신자가 답할 수 있는 것**(대상 앱이 클로드냐 코덱스냐, 어느 대화방인지, 무엇이 궁금한지 자체가 빠짐)이면 **사용자에게 되물어** 좁힌다. ⓑ **발신자·지금 방은 몰라도 대상 방은 알 법한 것**(예: "A사랑 비교했던 그 결제사 이름이 기억 안 나" — 그 이름은 대상 방 기록에 있음)이면 되묻지 말고 **서술적으로, 범위를 좁히지 말고 열어서 지칭해**("A사와 함께 비교했던 다른 결제사(들)의 장점") 넘긴다 — 그래야 대상 방이 하나로 못박지 않고 해당하는 걸 다 답한다. ⚠️ 이 괄호 속 예시는 설명용이다 — **예시 속 이름·문구를 실제 질문에 옮겨 적지 않는다**. 질문에는 사용자가 실제로 준 정보와 이 방의 실제 맥락만 넣는다(예시 누출 실측: 사용자가 말한 적 없는 회사명이 질문에 끼어 들어감). 대상 방은 자기 기록 전체를 아는 방이라 발신자가 이름을 몰라도 된다 — 이게 핫라인의 핵심 가치다. "그때 그거"처럼 발신자도 대상 방도 특정 못 할 때만 사용자에게 되묻는다.
   - **정돈이 질문 자체를 바꾸는 수준**이면 보내기 전 "이렇게 물어볼게: 「…」 맞아?" 한 줄로 확인받는다(§2의 *대화방* 확인과 별개인 *질문 문구* 확인 — 단순 구체화면 생략 가능, 남용해 초보를 지치게 하지 않는다).
   - 사용자가 "그대로 보내줘"라고 하면 정돈 없이 원문 그대로.

2. **감싼다**(자연스럽게, 명령조·과잉 지시 금지 — 검증사실: 주입조는 거부되고, 과잉 안내는 자연스러운 회신을 오히려 망친다):

⚠️ 첫 줄의 두 슬롯은 **둘 다 발신(나) 기준**이다 — 슬롯 이름 그대로: `<발신 앱(나)>` = 내가 속한 앱(Claude Code 또는 Codex), `<발신 대화방 제목(나)>` = 질문을 보내는 **내 방** 제목. **대상 방의 앱·제목은 절대 여기 넣지 않는다**(혼동 실측 — 발신처가 왜곡되면 수신 방이 출처를 오해한다. 대상 방 이름은 첫 줄에 아예 등장하지 않는 게 정상이다).

```text
<발신 앱(나): Claude Code|Codex> "<발신 대화방 제목(나)>" 대화방에서 질문이 도착했어요. 현재 대화방의 맥락을 근거로 답해 주세요. (핫라인 재호출 등 다른 대화방 호출은 하지 마세요.)

<정돈한 질문>
```

- 내 제목: `resolve claude "$CLAUDE_CODE_SESSION_ID"`로 조회. 자동 생성 제목이라 안 나오면 그냥 "다른 Claude Code 대화방"이라고 쓴다. **jsonl UUID를 발신 주소로 적지 않는다**(아래 ⚠️).

- **보고형 메시지**(답장 불필요 — 피드백 전달·상태 공유)면 템플릿을 그에 맞게 변형한다: 첫 줄을 "…에서 온 참고용 보고입니다(답장은 필요하면만)"로, 회신 레시피(§4B)는 "답장이 필요하면 …" 조건부로. 재귀 금지·도구 혼동 방지 문구는 어떤 형이든 유지.

3. **감싼 전문**을 임시파일로: `Q=$(mktemp -t hotline.XXXXXX)` 에 저장(repo 밖 — 실수 커밋 방지). 전문은 파일·stdin으로만 전달되므로 `-`로 시작하는 질문이 플래그로 오인될 여지도 없다.
4. `bash "$HOTLINE" scan "$Q"` — rc=1이면 **차단하고 사용자에게 보여준 뒤** 제거/진행 확인(질문은 대상 세션 로그에 영구 기록되므로). `.env` 같은 산문 언급도 걸리는 안전 쪽 오탐 설계 — 오탐이어도 사용자 확인은 생략하지 않는다.

## 4. 전송

### 4A. CLI 경로 (동기 — 답이 바로 돌아옴)

```bash
# 대상=Codex (어디서든). HOTLINE_CONFIRM=1은 §2 echo-back 확인에서 사용자 확인을 받은 뒤에만 붙인다
# (구조적 가드 — 대상 원본 기록에 append되는 비가역 동작이라 플래그 없이는 스크립트가 거부):
HOTLINE_CONFIRM=1 bash "$HOTLINE" ask codex "<ID>" "$Q"
# 대상=Claude (나=Codex일 때; cwd는 resolve 결과의 cwd. fork라 원본 미변경 — 플래그 불필요):
bash "$HOTLINE" ask claude "<UUID>" "<cwd>" "$Q"
```

- 답 = stdout. 끝나면 `rm -f "$Q"`.
- 실패(rc≠0)는 스크립트가 이유를 뱉는다 — **그대로 사용자에게 보고**(성공 위장 금지). rc=142=타임아웃(기본 300초, `HOTLINE_TIMEOUT`으로 조정).
- 안전장치(스크립트 내장): Codex=read-only 샌드박스(파일쓰기 차단·실측)+approval_policy=never / Claude=`--fork-session`(원본 미변경)+도구 allowlist+MCP 차단 / 재귀 sentinel / 열린 세션 경고.

### 4B. send_message 경로 (나=Claude → 대상 Claude — 비동기)

1. `list_sessions`로 대상 찾고 echo-back 확인(§2).
2. send_message 경로 전용 템플릿 — **미니멀하게**. 받는 쪽 화면엔 정확한 답장 주소가 담긴 "From …" 라벨이 자동으로 붙고, 모델은 그걸 보고 알아서 같은 경로로 회신한다(실측 다수). 안내는 도구 이름 한 줄이면 충분:

```text
Claude Code "{내 대화방 제목}" 대화방에서 질문이 도착했어요. ccd_session_mgmt의 send_message 도구(안 보이면 ToolSearch로 로드)로 이 메시지의 발신 세션(From 라벨)에 답장을 보내 주세요.

{질문 내용}

* 참고: 서브에이전트용 "SendMessage" 도구가 아님. 핫라인 재호출 금지.
```

- ⚠️ **발신 id를 본문에 절대 적지 않는다** — 특히 jsonl UUID(`$CLAUDE_CODE_SESSION_ID`)는 send_message의 `local_…` 주소 체계와 **다른 체계**라, 받는 쪽이 그 id로 답장을 시도하면 "존재하지 않는 세션"으로 실패한다(실사고 2026-07-06). 정확한 주소는 harness가 붙이는 From 라벨에만 있다 — 그걸 신뢰하게 둔다.
3. 사용자에게는 "답장은 이 대화방으로 비동기로 도착한다"고 안내(자동 라벨 "From {제목}"도 붙는다).
4. **정직 보고**: "전송됨(사용자 승인 시). 회신은 비동기로 도착 — 매 전송은 사용자 승인 게이트를 거치고, 상대 세션이 회신하지 않을 수도 있다. 안 오면 알려줘." **세 요소(비동기 도착 · 승인 게이트 · 회신 비보장)를 빼먹지 말고 전부 전한다.** 회신이 안 왔는데 온 것처럼 요약하지 않는다.

## 5. 답 제시 + 정리

- **원문 답은 보존하되, 그 위에 초보가 판단할 수 있는 쉬운 정리를 얹는다**(핵심 — README가 약속한 "정리해줘요"의 실체): ⓐ 대상 세션의 답을 그대로 전하고(요약하면 요약임을 밝힘 — 원문을 각색·왜곡해 덮지 말 것, **원문의 숫자·고유명사·명령어·파일 경로·URL·환경변수·id는 그대로 옮기고, 원문에 없는 예시·수치를 지어내 보태지 말 것**. 원문에 여러 항목·후보가 있으면 빠뜨리지 말고 다 담는다 — 하나로 뭉개지 말 것) ⓑ 그 아래 "그래서 …라는 뜻이고, 결정하려면 …" 식으로 사용자가 바로 판단·다음 행동할 수 있게 풀어준다 — **해설도 원문에서 논리적으로 따라오는 것만, 새 사실·수치·비교를 사실처럼 도입 금지**. 받은 답이 여러 후보 중 하나만 고른 듯하면 그 사실도 사용자에게 알린다. 대상 세션이 거부·빈 답이면 그대로 보고(억지 정리 금지). **보고 첫머리에 방식 명시**: "사본(fork) 자문 완료(원본엔 안 보임)" / "원본 전달됨(회신은 비동기)" / "Codex 자문 완료(내부 기록에 남음 — 앱 화면엔 재시작 후 보일 수 있음)".
- **`ask claude`(fork) 보고는 검증범위를 분리(가짜 성공 방지 — 2026-07-06)**: "fork 자문 완료 — 대상 세션 기록을 읽은 fork가 *이번 입력*에 답함. **미검증**: 원본 화면 도착 / 원본 기록 저장 / 다음 호출의 기억 / 왕복." **답변의 "보인다/기억난다"를 원본 전달·연속성 증거로 옮겨 적지 마라** — fork가 현재 입력 또는 원본의 기존 맥락을 해석한 결과일 수 있다.
- 임시파일 삭제: `rm -f "$Q"` — 이건 mktemp로 만든 내 파일을 지우는 정규 절차다(파일 삭제 확인 규칙과 충돌 아님).

## Troubleshooting (실사용 사례 기반)

- `HOTLINE-WARN`은 **비치명**(경고만, 실행 계속) — 이게 뜬 뒤 실패했다면 실패 원인은 다른 데 있다.
- `HOTLINE_CONFIRM=1`은 **ask codex 전용**(append 승인 게이트) — ask claude엔 아무 효과 없다.
- Codex 발신에서 `ask claude`가 rc=1이면: **발신측 sandbox가 `~/.claude` 접근을 막았을 가능성부터** 의심 — sandbox 밖 실행(approval)으로 재시도(실측: 이 방법으로 성공).
- Codex 발신에서 `ask codex`도 sandbox 안에서 실패할 수 있다(`~/.codex/state_*.sqlite` readonly 오류 — 실측 2회) — 마찬가지로 sandbox 밖 실행으로 재시도.
- "기록에 남는다는데 대상 Codex 앱 화면에 안 보여요": 내부 기록엔 있다 — 앱이 외부 append를 즉시 반영 안 함. **앱을 완전히 껐다 켜면 보인다**(실측 2회 재현).
- 이름해석이 0건인데 세션이 분명 있으면: 제목 오타·이모지 누락, 또는 customTitle이 없는 세션(→ id 직접지정).

## 한계 — 정직 표기 (숨기지 않는다)

- **read-only의 범위(정확히 — 과장 금지)**: Codex read-only 샌드박스는 **파일 쓰기를 막는다(실측 확인: 쓰기 시도 BLOCKED)** + `approval_policy=never`로 승인 탈출까지 차단. 단 **대상 세션에 연결된 MCP 커넥터·네트워크 접근까지는 완전히 닫지 못한다**(Claude 경로의 `--strict-mcp-config`와 달리 Codex 경로는 도구면을 전부 차단하진 못함 — 실측: read-only exec에서도 MCP 연결 시도 관찰). 읽기·응답 생성·토큰 비용도 막지 않는다. → 외부 쓰기 도구(커넥터)가 붙은 아주 민감한 세션은 자문을 신중히. Claude 대상(fork)은 `--strict-mcp-config`로 도구면까지 닫힘. Codex 대상은 자문 내용이 원본 세션 기록에 **append**된다(fork 불가 — 실측).
- **시크릿 스캔은 best-effort 안전망(보장 아님)**: 알려진 키 형식(OpenAI·Anthropic·AWS·GitHub·GitLab·Slack·Google·Stripe·npm·JWT·DB URL·token/session/cookie 등)을 순수 grep으로 잡는다(외부 도구 무설치). **미지의/사내 커스텀 형식·인코딩된 시크릿은 못 잡을 수 있다** — 최종 판단은 사용자 몫, 질문에 시크릿을 애초에 안 넣는 게 최선. 스캔이 파일을 못 읽으면 통과가 아니라 **중단**(F1).
- **`ask claude`는 원본 전달이 아니다**: read-only 분신(fork)이 답하며 원본 세션은 이 대화를 모른다. 원본 Claude 세션에 실제로 메시지를 넣는 경로는 send_message뿐이고, 이는 Claude Code 발신에서만 가능(Codex엔 해당 MCP 없음 — 정직 한계).
- **"어디에 남나"는 3층으로 구분해 말한다** (혼동 사건 2건의 교훈): ①내부 기록 파일에 남나 ②그 대화방의 이후 맥락에 포함되나 ③사용자 앱 화면에 보이나 — 셋은 다르다. Codex 대상 CLI: ①✅ ②✅ ③앱 재시작 후(즉시 아님, 실측). 코덱스→코덱스 native(`send_message_to_thread`): ①②③ 전부 ✅(즉시). Claude 대상 CLI(fork): ①②③ 전부 ❌(분신만 알고 사라짐). Claude 대상 send_message: ①②③ 전부 ✅.
- **연속 대화(핑퐁) 가능 여부**: Codex 대상은 같은 대화방 기록에 쌓여 후속 질문이 이전 문답을 기억한다(이어짐). native 경로 2종(클→클 send_message, 코→코 send_message_to_thread)은 진짜 왕복 협업 가능. Claude 대상 CLI는 매번 새 분신이라 **1회성**(후속 질문은 이전 문답을 모름) — 이어받기는 v1.1 후보.
- **재귀 차단**: CLI 경로는 env sentinel로 하드 차단. send_message 경로는 프롬프트 문구가 유일한 방어(env 미전파).
- **send_message 회신은 비보장**: 회신엔 ⓐ 수신 세션이 회신을 *결정*하고 ⓑ 사용자가 그 회신 전송을 *승인*하는 두 조건이 필요하다 — 둘 중 하나라도 빠지면 회신이 안 올 수 있다는 뜻(이게 "비보장"). **단 도달·깨우기 자체는 확실**: 닫힌·dormant(과거) 세션에도 도달해 깨우고 응답시킨다(실측 2026-07-07 — 3일 전 세션·과거 세션 모두 깨어나 회신).
- **큰 세션 비용**: 12MB 세션 15초 실측(hang 없음). 단 대형 컨텍스트 추론 요금이 든다 — 아주 큰 세션이면 사용자에게 미리 언급.
- **macOS 전용 + 비공식 내부 포맷 의존**: `~/.claude/projects`·`customTitle`·`session_index.jsonl`·resume 플래그는 공식 API가 아니다. CLI 업데이트로 깨질 수 있고, 그때 doctor가 잡는다. 검증 기준: 2026-07-11 · claude 2.1.206 · codex 0.144.0-alpha.4.
- **customTitle 없는 Claude 세션**은 이름으로 못 찾는다 — id로 우회. **다른 컴퓨터의 세션**은 범위 밖(세션 파일은 로컬 저장).
- **이름해석 성능**: `resolve claude`는 모든 세션 파일을 훑는다(마지막 customTitle 판별에 구조상 필요) — 세션이 많고 크면 수 초 걸릴 수 있다. 급하면 id 직접지정이 빠르다.
